## What Works

- **`formatCost` 3 sig figs for sub-cent** ‚Äî use `Math.floor(Math.log10(usd))` to get the leading digit's magnitude, then `dp = -magnitude + 2` for exactly 3 significant figures. Don't strip trailing zeros for sub-cent amounts ‚Äî they're significant. Only strip for `$0.01‚Äì$0.99` range.

- **Hover popovers must portal to `document.body` + `position: fixed`** ‚Äî the new generic `HoverCard` (`vendor/ui/kit/HoverCard.tsx`) measures the trigger via `getBoundingClientRect()` and renders the card through `createPortal`. Absolute positioning inside the PR-list table / timeline rows gets clipped by row `overflow`, so don't reach for the click-based `Dropdown` (which uses `position:absolute`) for hover previews. Close on a ~120ms delay (on both trigger and card `mouseleave`, cleared on re-enter) so the pointer can travel into the card to click the `file:line` links; close on scroll/resize to avoid a stale fixed position.

- **Line-numbered "code editor" = disable soft-wrap, sync the gutter's `scrollTop`** ‚Äî `vendor/ui/kit/CodeTextarea.tsx` puts a fixed-width number gutter next to a real `<textarea>`. The trap is soft-wrap: a wrapped logical line occupies 2+ visual rows and the 1:1 gutter numbering desyncs. Fix is `wrap="off"` + `whiteSpace:"pre"` on the textarea (one logical line = one visual row; long lines scroll horizontally), and mirror the textarea's vertical scroll onto the gutter via `onScroll ‚Üí gutterRef.scrollTop = ta.scrollTop`. Gutter + textarea must share identical `fontSize`/`lineHeight`/top-padding or the numbers drift. It stays a controlled `<textarea value/onChange>`, so `getByDisplayValue` RTL tests keep working ‚Äî the ConfigTab test needed no change when swapping `Textarea`‚Üí`CodeTextarea`.

## What Doesn't Work

- **`PrDetail.files` (from `GET /pulls/:id`) is NOT guaranteed unique per `path` ‚Äî keying diff cards by `f.path` crashes React on some PRs.** `pr_files` has no unique `(pr_id, path)` constraint, and at least one imported PR (PR #2, "Conventions Extractor‚Ä¶") carries TWO rows for the same file (`CLAUDE.md` as `+1/-0` AND `+0/-61`; same for `client/CLAUDE.md`) ‚Äî separate diff fragments the importer stored as distinct rows. Both the shared `diff-viewer/DiffViewer` and the new `SmartDiffViewer`/`FileCard` used `key={f.path}`, so those PRs threw `Encountered two children with the same key` at render (React may then drop/duplicate cards). Not caught by any test because fixtures use unique paths, and not seen on most PRs. Fix (composition layer, NOT a `key={path+index}` hack ‚Äî a file's diff belongs in one card): `dedupeFilesByPath` in `diff-viewer/helpers.ts` merges rows per path (sums +/- stats, concatenates the `@@` hunk fragments), called once in `DiffTab` before either viewer; the server `modules/smart-diff/service.ts` merges by path too so grouped-view data is unique. Before using any server-provided list as React keys, dedupe it ‚Äî the DB/contract doesn't promise uniqueness.

- **Adding `useTranslations("<new-namespace>")` inside a widely-shared component silently breaks every existing caller that doesn't register that namespace in its `NextIntlClientProvider`.** `diff-viewer/FileCard.tsx` is consumed both by the plain `DiffViewer` (existing callers, `shell` namespace only) and the new `SmartDiffViewer` (needs `smart-diff` strings for the findings badge / mechanical placeholder). Calling `useTranslations("smart-diff")` unconditionally inside `FileCard` doesn't fail typecheck or most tests, but throws `IntlError: MISSING_MESSAGE` at render for any consumer whose provider doesn't include that namespace ‚Äî caught only by `src/test/smoke.test.tsx`'s narrower `{shell: shellMessages}` provider logging the error to stderr (test still "passed" since next-intl swallows it and renders a fallback, so this is easy to miss). Fix: keep the shared component locale-agnostic beyond its original namespace ‚Äî accept **pre-formatted strings** as props (`findingsLabel?: string`, `mechanicalPlaceholderText?: string`) with a plain-English fallback (`?? `${n} findings``), and do the actual `useTranslations("smart-diff")` call one level up, in the feature-specific caller (`SmartDiffViewer`) that already needs that namespace. Generalizes: before adding a new `useTranslations(ns)` call to a component with more than one caller, check every caller's `NextIntlClientProvider messages={...}` (tests especially) for that namespace, or push the translation call up to the caller instead.

- **A single flex-wrap row mixing a title with secondary metadata (edit icon, tag) gives inconsistent gaps depending on title length.** `ConventionCandidateCard`'s header originally put the rule title, an edit `IconBtn`, and a category tag all in one `flexWrap: "wrap"` container with a uniform `gap`. It looked fine for a short one-line title but looked cramped for a long title that wrapped to 2 lines ‚Äî the metadata row that wrapped underneath only got the same small inline `gap` as spacing from the paragraph above, not a real margin. Fix: don't rely on wrap behavior for visually-distinct rows ‚Äî split into an explicit column (`titleCol` ‚Üí `titleRow` + `metaRow`, each its own flex row) with a real `gap` between them, so spacing is consistent no matter how many lines the title takes.

- **A capture-phase `window` scroll listener that closes a popover will block scrolling INSIDE it** ‚Äî `HoverCard` closes on page scroll to avoid a stale `position: fixed`, but with `addEventListener("scroll", ‚Ä¶, true)` the handler also fires for wheel-scrolls of the card's own inner list, instantly closing it. Guard with `if (cardRef.current?.contains(e.target)) return;` so only page scrolls close the card.

- **`ConventionCandidateCard`'s evidence `path:line` is rendered via `MonoLink` with NO `href`** ‚Äî it degrades to a plain non-clickable button showing text only. `client/src/lib/github-urls.ts#githubBlobUrl` already exists and is used by `FindingCard`/`FindingsPopover` to build a clickable GitHub blob+line URL for PR-review findings, but nothing wires the same helper into the conventions candidate card. If the acceptance criterion is "clicking a candidate's evidence opens the real file on GitHub," this currently does NOT work ‚Äî pass a `githubBlobUrl(...)` result as `MonoLink`'s `href` (needs repo owner/name + ref, not just path/line, which the candidate's evidence fields don't currently carry ‚Äî check what `ConventionCandidate` needs added).

- **`ConventionCandidateCard`'s left border is intentionally always `var(--ok)` (green), NOT status-driven** ‚Äî `styles.ts`'s `card()` used to color it by status (`accepted ‚Üí var(--ok)`, `rejected ‚Üí var(--text-muted)`, `pending ‚Üí var(--border-strong)`), which looks like the "obviously correct" design (status at a glance) and matches how the Accept/Reject buttons still work (kind switches to `primary`/`secondary` per status). Explicit user design call: keep the border always green regardless of status ‚Äî don't reinstate the status-color branching, that reverts a deliberate decision, not a bug fix. `card()`'s signature dropped its `accepted` param accordingly (now just `card(rejected: boolean)`, `rejected` still drives the dimmed opacity).

## Codebase Patterns

- **`FindingsPopover` is shared by the PR list and the detail timeline** ‚Äî it lives at `app/repos/[repoId]/pulls/_components/FindingsPopover/` (the parent `pulls/` route, so both the list `page.tsx`/`PRRow` and the nested `[number]/_components/RunHistory` import it). It takes `findings: Finding[]`; `FindingRecord[]` (timeline reviews) is structurally compatible, so pass it directly. The timeline's per-run findings come for free from the already-loaded `usePrReviews` data ‚Äî `FindingsTab` builds `findingsByRun` (`run_id ‚Üí review.findings`) and threads it into `RunHistory`, which falls back to the old "N findings ¬∑ M blockers" text when a run has no matching review (failed/running).

- **To make a whole row a hover trigger but anchor the card elsewhere, separate the hover area from the anchor** ‚Äî `HoverCard` supports `block` (wrapper is a full-width `div`, so an entire timeline row is the hover area) + `anchorRef` (a ref to a child, e.g. the severity badges, used for `getBoundingClientRect` positioning). The timeline's `RunRow` (extracted so each row can own a `badgesRef`) wraps the whole row and points `anchorRef` at the badges. NOTE: you can't show a hover-highlight by tinting the `block` wrapper ‚Äî the row's own `--bg-elevated` is opaque and sits on top; highlight the row element itself if needed. The non-`block` (inline `span`) mode keeps the plaque highlight for the PR-list badges.

- **`AppFrame`'s `<main>` (`vendor/ui/shell/AppFrame.tsx`) has zero padding by design** ‚Äî every top-level page view is responsible for its own page-level padding, not the shell. The convention (see `AgentsListView`'s `s.page: { padding: "24px 32px 44px", maxWidth: 1100, margin: "0 auto" }`) is to wrap the page's root JSX in a `page` style from the view's own `styles.ts`. `ConventionsView` was scaffolded without this wrapper, so its content rendered flush against the viewport edge (no gutters) even though its layout otherwise matched sibling pages ‚Äî easy to miss because nothing errors or looks obviously broken until compared side-by-side with another page. When adding a new top-level page under `app/**/_components/<View>/`, copy the `s.page` wrapper pattern rather than assuming `AppShell`/`AppFrame` supplies it.

- **`ConventionsView`'s candidate list is a persistent checklist, not a processing queue ‚Äî accepted/rejected cards deliberately stay visible, never filtered out.** `list.map(...)` renders every candidate regardless of `status`; only at "Create skill" time does `acceptedCandidates = list.filter(c => c.status === "accepted")` get pulled out for the modal. This is intentional per `docs/conventions-extractor-plan.md`'s Track 1 description ("approve / reject / edit / re-scan / **deselect-all**") ‚Äî the "Deselect all" toolbar button bulk-resets every non-pending card back to `pending`, which only makes sense if accepted/rejected cards remain visible and re-toggleable up until you actually generate the skill. Don't add filtering/hiding for accepted or rejected candidates in the main list ‚Äî that would break the ability to review/reverse decisions via Deselect all (or per-card, by re-clicking Accept/Reject) before finalizing.

## Tool & Library Notes

- **Evenly-spaced round dotted underline** (severity-coloured leader under the findings badges) is a `radial-gradient(circle, COLOR 1.1px, transparent 1.6px)` with `backgroundSize: "5px 2px"`, `backgroundRepeat: "repeat-x"`, `backgroundPosition: "left bottom"` ‚Äî NOT `border-bottom: dotted`, whose dots render as chunky touching squares. Set `background: transparent` first (bare badge) then the gradient via `backgroundImage` (later key wins over the shorthand's image reset). See `SeverityBadge underline` in `vendor/ui/primitives/Badge.tsx`.

- **`Icon` names are the map keys in `vendor/ui/icons.tsx`, NOT the raw lucide export names.** The map aliases some: e.g. `Edit: Pencil` ‚Äî so the valid `IconName` is `"Edit"`, and `<Badge icon="Pencil">` fails typecheck (`TS2322`). When picking an icon, grep the object keys in `icons.tsx` (the `satisfies Record<string, LucideIcon>` block) rather than assuming lucide's name is exported.

- **`Donut` (`vendor/ui/charts/Donut.tsx`) defaults its legend to a dollar amount** (`${valuePrefix}${n.toFixed(2)}`, `valuePrefix="$"`). For integer/count legends (e.g. findings-by-category), pass `format={(n) => String(n)}` ‚Äî a `format?: (n:number)=>string` prop overrides the money formatting without touching existing cost callers.

- **`Button`'s `active` prop only has a visual effect for `kind="tertiary"`** (`vendor/ui/primitives/Button.tsx`) ‚Äî the `secondary`, `ghost`, `primary`, and `danger` style maps never read `active` at all, so `<Button kind="secondary" active={someBoolean}>` silently renders identically whether `someBoolean` is true or false. `ConventionCandidateCard`'s Accept/Reject buttons passed `active={accepted}`/`active={rejected}` with `kind="secondary"`/`kind="ghost"` and had zero visual state feedback as a result (only the card's left border color changed). Fix is to switch `kind` itself based on state (e.g. `kind={accepted ? "primary" : "secondary"}`) rather than relying on `active` ‚Äî or extend the style maps if you actually want `active` to do something for those kinds.

- **`ConventionCandidateCard`'s Accept/Reject buttons are intentionally asymmetric in color, not a leftover bug.** Accept escalates to `kind="primary"` when `accepted` (the only kind with a filled `var(--accent)`/blue background per `Button.tsx`'s style map), while Reject only ever toggles between `"ghost"` (not rejected) and `"secondary"` (rejected) ‚Äî it never reaches `"primary"`, so it never gets a colored fill, only a neutral gray outline/fill. Don't "fix" Reject to also turn blue when accepted-style-symmetry looks broken ‚Äî Accept is deliberately the only button that carries the accent color, since it's the path toward the primary next action (bundling into a skill); Reject is modeled as a neutral/muted state. (The card's left border, separately, is always `var(--ok)` green regardless of status ‚Äî see the entry below ‚Äî only `opacity` in `styles.ts#card(rejected)` changes with status.)

- **TanStack Query v5: a DISABLED query (`enabled: false`) sits in `status: 'pending'` forever ‚Äî gate loading UI on `isLoading`, not `isPending`.** `useQuery({ enabled: prId != null })` never runs while disabled, so `isPending` stays `true` indefinitely (there's no cached data). A component doing `const loading = query.isPending` then renders its skeleton FOREVER whenever the query is disabled ‚Äî the exact `prId === null` case an `enabled` guard is written to handle. Use `isLoading` (v5 = `isPending && isFetching`), which is `false` for a disabled query. Also relevant for mutations: don't fold `mutation.isPending` into the same body-`loading` flag or a re-fetch/recompute wipes the currently-shown data ‚Äî let the trigger button show its own `loading` spinner instead (see `IntentPanel.tsx`).

## Recurring Errors & Fixes

- **A skills mutation that creates a NEW version must invalidate `["skill-versions", id]`, not just `["skills"]` / `["skill", id]`.** `useUpdateSkill` invalidates only the latter two, so after a body change (which bumps the version and snapshots a new `skill_versions` row) the Versions tab's `useSkillVersions` query stays stale until some unrelated refetch. The dedicated `useRestoreSkill` (restore always creates a version) adds the `["skill-versions", data.id]` invalidation ‚Äî mirror that in any future mutation that can change `body`. Restore itself is now a real server endpoint (`POST /skills/:id/restore`), not the old client-side workaround of re-PUTting the old body via `useUpdateSkill`.

- **`GET/POST /agents/:id/skills` returns bare `AgentSkillLink` rows (`{ agent_id, skill_id, order }`), NOT a joined `{ skill: Skill, order }`.** It's easy to assume otherwise because the server-side `AgentsRepository.linkedSkills()` *does* return a joined `LinkedSkillRow` (`{skill, order}`) ‚Äî but the route (`agents/routes.ts`) calls `service.skillLinks()`, which maps to the public `AgentSkillLink` DTO (already in `@devdigest/shared`), i.e. ids only. A mocked-fetch hook test using the wrong (joined) shape will pass green while the real endpoint silently returns `undefined` for `.skill.id`/`.skill.enabled` at runtime ‚Äî caught only by hitting the live API (`curl localhost:3001/agents/:id/skills`), not by typecheck or the component's own RTL test. If you need the full `Skill` for a link row, look it up by `skill_id` against an already-fetched skill list (e.g. `useSkills()`) ‚Äî see `SkillsTab.tsx`'s `byId` map.

### 2026-07-08 ‚Äî Clickable Smart Diff severity chips ‚Üí deep-link to a finding in "Agent runs"
- **A URL-param cannot make a `FindingCard` "expand itself on mount" ‚Äî the card isn't mounted.** The findings tab nests `FindingsTab ‚Üí ReviewRunAccordion (own `open` state, collapsed by default except index 0) ‚Üí FindingsPanel ‚Üí FindingCard`. Deep-linking to a finding must therefore drive the *whole chain*: `ReviewRunAccordion` opens if `review.findings.some(f => f.id === target)`, then `FindingCard` expands + `scrollIntoView` via an effect keyed on the target. Mirrors the existing Timeline‚ÜíReview-run `targetRunId`/`targetNonce`+effect pattern (`ReviewRunAccordion.tsx:47`) rather than inventing a new mechanism.
- **`FindingsPanel`'s "hide low confidence" toggle can delete the navigation target from the DOM.** `shown = visibleFindings(findings, hideLow)` filters before mapping to cards, so deep-linking to a SUGGESTION/low-confidence finding while `hideLow` is on scrolls to nothing. Fix applied: when the target exists in `findings` but not in `shown`, `setHideLow(false)` first, then focus it. Any future "navigate to item X in a filtered list" needs the same guard ‚Äî reveal before scroll.
- **The severity chip is rendered in `CodeLine` but its finding identity is dropped one layer up.** `FileCard.severityByLine()` collapsed findings to `Map<lineNo, Severity>` (highest-severity-wins), discarding `f.id`. To make the chip a link it now carries `{severity, findingId}` (the *winning* finding's id). `CodeLine`/`FileCard` are shared with the plain `DiffViewer`, so the click is opt-in via an `onFindingClick?` prop (plain viewer passes none ‚Üí chip stays a static `<span>`), and no `useTranslations` was added there (aria-label is a hardcoded string, matching the existing "Add a comment on this line" label) ‚Äî keeps the shared components locale-agnostic per the FileCard rule above.
- **React treats `borderColor` (and `borderWidth`) as SHORTHANDS relative to `borderLeftColor`/`borderLeftWidth` ‚Äî being "mostly longhand" isn't enough.** `FindingCard/styles.ts`'s `s.card` was written with an explicit comment claiming it avoided the shorthand/longhand mix, yet it paired `borderColor` with `borderLeftColor` (and `borderWidth` with `borderLeftWidth`). React only fires the "Updating a style property during rerender (borderColor) when a conflicting property is set (borderLeftColor)" warning when the shorthand member actually *changes between renders* ‚Äî here when `focused` toggles the border color ‚Äî which is why it stayed silent until a feature (Smart Diff chip deep-link) started flipping `focused` on navigation. Fix: go FULLY per-side (`borderTop/Right/Bottom/LeftColor` + matching `...Width`), never `borderColor`/`borderWidth` alongside a `borderLeft*`. A latent version of this can hide in any card/row style that has a colored left-accent border plus a focus/hover color change. Caught only in the browser console (screenshot from the user) ‚Äî jsdom/vitest doesn't surface it, so component tests stayed green through the bug.

### 2026-07-09 ‚Äî `useEffect` that calls its own `setState` cancels its own cleanup timer (FileCard highlight never fades)
- **`FileCard.tsx`'s finding-scroll effect sets a 1500ms `setTimeout` to restore the row background, then calls `setPendingScrollLine(null)` in the same effect body ‚Äî but `pendingScrollLine` is in the effect's dep array, so that state write re-invalidates the effect and React runs its cleanup (`clearTimeout`) on the very next render, cancelling the restore.** Net result: the "transient" highlight applied via direct DOM mutation (`el.style.backgroundColor = 'var(--accent-bg)'`) is never removed and stays for the life of the card. Compounded by the fact that the highlight is written on the outer `rowWrap` div, whose React `style` prop (`cs.rowWrap`, a stable const) never sets `backgroundColor`, so React never overwrites it either. Surfaced by the pr-self-review react lens (conf 85); it's pre-existing base smart-diff code, not the clickable-badge change. Fix pattern: don't reset the trigger state inside the same effect ‚Äî track "consumed" via a `useRef`, or do the reset inside the `setTimeout` callback after restoring styles, so the effect isn't torn down by its own write. General rule: an effect must not `setState` on a value it depends on unless it wants to re-run + re-clean-up itself.
