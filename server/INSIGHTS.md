Append-only log of decisions, gotchas discovered, and lessons for the API package.
Newest first. Capture what wasn't obvious from the code — the "why" behind a choice,

## What Works

- **`ReviewOutcome.costUsd` is the actual OpenRouter charge** — `reviewer-core/src/llm/openrouter.ts` extracts `usage.cost` from the API response and accumulates it across chunks. Always prefer this over `priceBook.estimate()` which uses static list prices and can be 15–35% off from the actual billed amount due to dynamic provider routing.

## What Doesn't Work

- **`priceBook.estimate()` for displayed cost** — Uses `/models` list price; OpenRouter routes dynamically (DigitalOcean, Cloudflare, Alibaba) with different prices per provider, plus caching discounts. Don't use as a display value — use only as a fallback when `costUsd` is null (old rows before the `cost_usd` DB column was added).

## What Doesn't Work

- **Splitting parallel agent work by file ownership isn't enough when tasks share global runtime state.** Building the Skills feature, one parallel task owned `server/src/modules/skills/**` (its own integration tests) and a separate task owned `server/src/db/seed.ts`/`seed-prompts.ts` (adding the seed data). They looked file-disjoint, but EVERY `*.it.test.ts` calls `seed()` in `beforeAll` — so the skills-module task's tests transitively depended on `seed.ts` being valid, and both tasks ended up editing `seed.ts`/`seed-prompts.ts` concurrently to unblock their own test runs. It resolved fine this time (final state was consistent, verified by rerunning typecheck + all test suites), but it was luck of write-order, not design. When splitting work across agents/sessions in this repo, treat `seed.ts` as a shared resource requiring sequencing, not something safe to touch from two lanes at once just because the primary target files differ.

## Tool & Library Notes

- **fflate `unzipSync` — enumerate before you inflate (decompression-bomb defense).** The `filter` callback runs per entry BEFORE decompression and receives `{ name, size, originalSize, compression }`, where `originalSize` is the uncompressed size from the zip header. Returning `false` skips inflating that entry entirely. So the safe pattern for reading one file out of an untrusted `.zip` (see `modules/skills/helpers.ts`) is TWO passes: pass 1 `filter: () => { infos.push(...); return false }` collects names + declared sizes with zero inflation → cap entry-count and the target's `originalSize`; pass 2 `filter: (f) => f.name === target` inflates only the chosen entry. A lying (understated) size header can't slip past — fflate allocates `out` to the declared size and errors mid-inflate when the stream overruns it, so a `try/catch` at the caller still bounds it. Don't call bare `unzipSync(buf)`: it eagerly inflates ALL entries into memory synchronously and blocks the event loop.

## Recurring Errors & Fixes

- **Committing a shared file that's already dirty with unrelated uncommitted work sweeps that work into your commit too** — `git add <path>` (even a pathspec-limited `git commit -- <path>`) stages/commits the file's WHOLE current diff, not just the hunk you just typed. Hit this 3× building the Conventions Extractor on a branch that also had a separate, unfinished "skills" feature sitting uncommitted in the same working tree: `server/src/modules/index.ts` and `server/src/db/rows.ts` already had uncommitted skills-module registration lines before the conventions task touched them (client's `nav.ts` had the identical problem — see `client/INSIGHTS.md`). Fix pattern: `git show HEAD:path/to/file > /tmp/clean.ts`, hand-edit `/tmp/clean.ts` to add ONLY your change on top of real HEAD content, `cp /tmp/clean.ts path/to/file && git add path/to/file` and commit that — then re-apply your original full edit (your change + the pre-existing WIP together) to the working tree afterward, uncommitted, so `git diff HEAD -- path/to/file` shows only the pre-existing WIP residual and the file stays fully correct for local testing. Verify with `git show --stat <commit>`, not just `git status` after the fact; `git reset --soft HEAD~1` + redo is the recovery move if caught late.

- **A `FeatureModelId` registry entry can default to a provider you have no key for.** `FEATURE_MODELS` (`vendor/shared/contracts/platform.ts`) hardcodes a default `provider`/`model` per feature — e.g. `conventions` → `openai`/`gpt-5.4` — independent of what's actually in `~/.devdigest/secrets.json`. A partial local setup may only have `OPENROUTER_API_KEY`, so calling an unoverridden feature fails at call time with `{"error":{"code":"config_error","message":"OPENAI_API_KEY is not configured"}}`. Fix: `PUT /settings` with `{"feature_models": {"<id>": {"provider": "openrouter", "model": "<a real openrouter model>"}}}` before exercising that feature. Resolution goes through `resolveFeatureModel(container, workspaceId, id)` (`modules/settings/feature-models.ts`), which falls back to the registry default only when no override row exists. A `routeModel(...)` helper is referenced in at least one planning doc (`docs/conventions-extractor-plan.md`) but does **not exist** in this codebase.

- **A new `NOT NULL text` column with no `DEFAULT` on an existing table breaks the migration the moment the table has any rows** — drizzle-kit happily generates a bare `ALTER TABLE ... ADD COLUMN category text NOT NULL` (no default) if the schema field is `text('category').notNull()` with nothing else; it applies fine against an empty dev table but fails on any environment where the table is already seeded. `status` on the same table shipped correctly (`.notNull().default('pending')`) — always add a `.default(...)` (or split into add-nullable → backfill → set-not-null across migrations) for any `NOT NULL` column added to a table that might already have rows.

- **API serving stale code despite edits → check for duplicate `tsx watch src/server.ts` processes.** Multiple `pnpm dev` instances can accumulate; only one binds `:3001` and the others' children may keep serving OLD code (or a killed watcher leaves an orphaned child on the port). Symptom: a route change has no effect even though the file on disk is correct. Fix: `lsof -nP -iTCP:3001 -sTCP:LISTEN` to find the real owner, `kill` all `tsx watch` parents + the listening child, confirm `:3001` is free, then start one fresh `pnpm dev`.

- **`docker exec devdigest-postgres psql -U postgres …` fails with `FATAL: role "postgres" does not exist`.** The container has no default superuser role — the DB and role are both `devdigest` (see `DATABASE_URL` in `server/.env` / `docker-compose.yml`). Use `psql -U devdigest -d devdigest`. Useful for ad-hoc checks like confirming an applied migration: `select id, created_at from drizzle.__drizzle_migrations order by id desc;`.

## Codebase Patterns

- **`ReviewRepository` (repository.ts) wraps `run.repo.ts`** — both files define `completeAgentRun`. When adding a new field to the values type, update BOTH files: `repository/run.repo.ts` (the implementation) and `repository.ts` (the wrapper used by `run-executor.ts`).

- **`GET /repos/:id/pulls` returns `Promise<PrMeta[]>` with NO response Zod schema** (only `params: IdParams`). Adding a field to `PrMeta` is therefore a pure type change — the object is serialized as-is, nothing strips unknown keys. The list popover's `latest_findings` (full `Finding[]`) is built in the SAME findings query that computes the severity counts (`fullFindingsByReview` alongside `findingsByReview`), so previewing findings on the list needs no extra request. `severity`/`category`/`kind` are `text` columns → cast to the `Finding` enum types (`as Finding['severity']`).

- **PR-list FINDINGS + SCORE aggregate across the latest review PER AGENT, not the single newest review** (`currentReviewIdsByPr` keyed by `prId:agentId`, newest-first → first seen wins). Re-runs create new `reviews` rows, so a PR that one agent re-ran clean while another still flags issues must still show those issues — "current open findings" semantics (product-confirmed: clean latest reviews ⇒ show "—"). PR `score` is the MIN across those current per-agent reviews (worst agent), so the score stays coherent with the findings shown. The `/pulls/:id/reviews` endpoint, by contrast, returns ALL review rows (full history) — so the detail page shows superseded findings the list intentionally omits.

- **The PR-list aggregation (score + findings + cumulative cost) lives in `modules/pulls/review-meta.repo.ts`, NOT inline in `routes.ts`.** `loadPrReviewMeta(db, priceBook, prIds)` runs the 3 IN-queries + JS grouping and returns `Map<prId, PrReviewMeta>`; the route handler just maps rows through it. Keep new list-aggregation logic there — `routes.ts` is intentionally thin (per `server/CLAUDE.md`: routes thin, data access in repositories). The route still does the GitHub sync + base PR select directly; that's pre-existing.

- **Run-cost fallback is centralized in `PriceBook.resolve(costUsd, model, tokensIn, tokensOut)`** — returns the actual API cost when stored, else `estimate(...)` for pre-migration rows, else null. Don't re-inline the `costUsd ?? estimate(...)` ternary; the list (`review-meta.repo.ts`), run history (`reviews/service.listRuns`), and trace (`reviews/service.getRunTrace`) all go through `resolve` so they stay consistent.

- **Drizzle transactions + row locking ARE available and now used** (first site: `modules/skills/repository.ts`). `this.db.transaction(async (tx) => { ... })` — do all writes on `tx`, return value commits, throw rolls back. For a read-modify-write on a counter (e.g. bumping `skills.version` + inserting the matching `skill_versions` snapshot), lock the row inside the txn with `.select()....where(...).for('update')` so concurrent edits serialize instead of both reading version N and colliding on `(skillId, version)`. Pattern to AVOID: reading via a non-locking helper (`getById`) then updating — that's the race. Also: `.for('update')` is drizzle's `SELECT ... FOR UPDATE`; prefer it over `onConflictDoNothing`, which silently swallows the lost write and hides the divergence.

### 2026-07-04 — Onion gate on an all-uncommitted feature branch (pr-self-review)
- **The gate config isn't durably present.** Both `dependency-cruiser.arch.cjs` and `known-violations.json` were **absent** this session despite the 2026-07-03 note claiming they were "added" — the config only lives in the skill folder and had to be re-bootstrapped verbatim from `onion-architecture/enforcement.md`. Don't assume the baseline exists before running the gate.
- **The `--output-type baseline` → `--ignore-known` workflow is a trap when the whole feature is uncommitted** (BASE==HEAD, i.e. everything is staged+unstaged+untracked, as on a fresh hw branch). Baselining the working tree snapshots YOUR OWN new violations into `known-violations.json`, so the gate then passes trivially and catches nothing. Instead: run depcruise **without** a baseline (`--output-type err`) and manually scope the reported edges to the changed files. Doing so this session confirmed the new `skills` module + `run-executor.ts` introduce **0** new boundary violations — all 17 reported are the pre-existing ones in unchanged files (`repo-intel`, `reviews/diff-loader`, Drizzle-in-routes, container cycles).

### 2026-07-04 — pr-self-review re-run found a previously "fixed" CRITICAL was not actually fixed
- **Patching a bad migration with a LATER migration doesn't fix it — the bad one still runs first.** Commit `ccba0dd` believed it fixed the `category text NOT NULL` no-default bug (see the "A new `NOT NULL text` column..." entry above) by adding `.default('uncategorized')` to `schema.ts` and generating a new migration `0012` containing only `ALTER COLUMN category SET DEFAULT 'uncategorized'`. It never regenerated `0011` itself, which still has the bare `ADD COLUMN category text NOT NULL` with no default. Since `0011` applies before `0012`, the deploy still fails on any non-empty `conventions` table — `0012`'s default never gets a chance to help. The correct fix for a not-yet-shipped migration is to delete the patch migration and **regenerate the original one** from the now-corrected schema, not bolt a follow-up `ALTER COLUMN` onto it.
- **A `pr-self-review` report's `outcome: "fixed"` / `fixCommit` annotation is a claim, not a proof — re-verify it on every re-run, don't just carry it forward.** The prior report (headSha `1d2bfb6`) recorded this bug as `"outcome": "fixed"` and the branch went on to a `"verdict": "pass"`. Two commits later, re-running the skill's deterministic migration-safety check (reading the actual `.sql` files, not trusting the old report) found the bug still live. Treat every prior "fixed" finding as a hypothesis to re-check against current file contents, especially for migration files, which are easy to "fix" at the wrong layer (schema vs. the already-generated SQL).
