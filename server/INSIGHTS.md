## What Works

- **`ReviewOutcome.costUsd` is the actual OpenRouter charge** ‚Äî `reviewer-core/src/llm/openrouter.ts` extracts `usage.cost` from the API response and accumulates it across chunks. Always prefer this over `priceBook.estimate()` which uses static list prices and can be 15‚Äì35% off from the actual billed amount due to dynamic provider routing.

## What Doesn't Work

- **`priceBook.estimate()` for displayed cost** ‚Äî Uses `/models` list price; OpenRouter routes dynamically (DigitalOcean, Cloudflare, Alibaba) with different prices per provider, plus caching discounts. Don't use as a display value ‚Äî use only as a fallback when `costUsd` is null (old rows before the `cost_usd` DB column was added).

## What Doesn't Work

- **Splitting parallel agent work by file ownership isn't enough when tasks share global runtime state.** Building the Skills feature, one parallel task owned `server/src/modules/skills/**` (its own integration tests) and a separate task owned `server/src/db/seed.ts`/`seed-prompts.ts` (adding the seed data). They looked file-disjoint, but EVERY `*.it.test.ts` calls `seed()` in `beforeAll` ‚Äî so the skills-module task's tests transitively depended on `seed.ts` being valid, and both tasks ended up editing `seed.ts`/`seed-prompts.ts` concurrently to unblock their own test runs. It resolved fine this time (final state was consistent, verified by rerunning typecheck + all test suites), but it was luck of write-order, not design. When splitting work across agents/sessions in this repo, treat `seed.ts` as a shared resource requiring sequencing, not something safe to touch from two lanes at once just because the primary target files differ.

## Tool & Library Notes

- **fflate `unzipSync` ‚Äî enumerate before you inflate (decompression-bomb defense).** The `filter` callback runs per entry BEFORE decompression and receives `{ name, size, originalSize, compression }`, where `originalSize` is the uncompressed size from the zip header. Returning `false` skips inflating that entry entirely. So the safe pattern for reading one file out of an untrusted `.zip` (see `modules/skills/helpers.ts`) is TWO passes: pass 1 `filter: () => { infos.push(...); return false }` collects names + declared sizes with zero inflation ‚Üí cap entry-count and the target's `originalSize`; pass 2 `filter: (f) => f.name === target` inflates only the chosen entry. A lying (understated) size header can't slip past ‚Äî fflate allocates `out` to the declared size and errors mid-inflate when the stream overruns it, so a `try/catch` at the caller still bounds it. Don't call bare `unzipSync(buf)`: it eagerly inflates ALL entries into memory synchronously and blocks the event loop.

- **`UnifiedDiff.files[]` has NO raw per-file `.patch` string** (`vendor/shared/adapters.ts:185`) ‚Äî `parseUnifiedDiff` (`adapters/git/diff-parser.ts`) already breaks each file into structured `DiffHunk[]` (`oldStart/oldLines/newStart/newLines/newLineNumbers`), discarding the raw `@@ ... @@` text. The raw `.patch` string only exists one layer down, on the `pr_files` DB row, consumed solely by `diff-loader.ts`'s reconstruction fallback (`diffFromPrFiles`) when no real clone is available. If you need `@@ -a,b +c,d @@` header text from an already-loaded `UnifiedDiff` (e.g. `modules/intent/signals.ts#hunkHeaders`, built for the cheap-signals PR-intent classifier), reconstruct it from the structured hunk fields ‚Äî `` `@@ -${h.oldStart},${h.oldLines} +${h.newStart},${h.newLines} @@` `` ‚Äî don't look for a patch substring to slice, it isn't there.

- **New optional fields on the `PromptAssembly` contract (`vendor/shared/contracts/trace.ts`) must be `.nullish()`, not `.nullable()`.** `PromptAssembly` has two constructors: `assemblePrompt` in `reviewer-core/src/prompt.ts` (sets every field, so it doesn't care) and `emptyPromptAssembly()` in `server/src/platform/trace-builder.ts`, which builds a literal that OMITS the optional fields (`callers`/`repo_map`/`pr_description`/`intent`). A `.nullable()` field is still a *required* key in the inferred TS type, so adding one silently breaks `emptyPromptAssembly()`'s literal at typecheck; `.nullish()` (optional + nullable) keeps that constructor compiling untouched. Confirmed while adding `intent` for the Intent Layer's `## Intent` prompt section ‚Äî mirror the change in BOTH the server and `client/src/vendor/shared/contracts/trace.ts` copies (byte-for-byte identical).

## Tool & Library Notes

- **`node:net`'s `BlockList` conflates address families when mixed rules share one instance.** A single `BlockList` that has BOTH a plain IPv4 subnet (`addSubnet('10.0.0.0', 8, 'ipv4')`) AND the IPv4-mapped IPv6 subnet `::ffff:0:0/96` (`addSubnet('::ffff:0:0', 96, 'ipv6')`) makes `check(anyPublicIpv4Address, 'ipv4')` return `true` for literally EVERY IPv4 address, not just ones in the listed ranges ‚Äî confirmed reproducibly on Node 22.23.1 with a throwaway `node -e` (`93.184.216.34` came back blocked even with no matching v4 rule). Node evidently normalizes an IPv4 check against the IPv4-mapped-IPv6 form too, so it collides with any `::ffff:0:0/96`-style v6 rule. Fix: use TWO separate `BlockList` instances ‚Äî one with only `'ipv4'`-typed rules, checked with `.check(ip, 'ipv4')` for `net.isIP(ip) === 4`; another with only `'ipv6'`-typed rules (including `::ffff:0:0/96` to catch IPv4-mapped IPv6 literals), checked with `.check(ip, 'ipv6')` for `net.isIP(ip) === 6`. See `adapters/http/web-fetch.ts` (`buildV4BlockList`/`buildV6BlockList`). Always verify a `BlockList`-based guard with a `node -e` sanity script covering a real public IP before trusting it ‚Äî this bug fails in the "blocks too much" direction (which looks safe but silently breaks the feature), the opposite failure mode from the fail-open regex bug it replaced.
- **The deprecated IPv4-COMPATIBLE IPv6 form `::a.b.c.d` (e.g. `::127.0.0.1`) is a distinct address space from the IPv4-MAPPED form `::ffff:a.b.c.d`.** Only `::/96` (all top 96 bits zero) catches the compat form; `::ffff:0:0/96` does not (its 80-95 bit group is `ffff`, not `0`). `::/96` also happens to cover the unspecified address `::` itself (`0:0:0:0:0:0:0:0` has all 96+ bits zero), so adding it closes both the `::` bypass and the `::a.b.c.d` bypass in one rule ‚Äî needed both in the SSRF guard's IPv6 `BlockList`.

## Recurring Errors & Fixes

- **Committing a shared file that's already dirty with unrelated uncommitted work sweeps that work into your commit too** ‚Äî `git add <path>` (even a pathspec-limited `git commit -- <path>`) stages/commits the file's WHOLE current diff, not just the hunk you just typed. Hit this 3√ó building the Conventions Extractor on a branch that also had a separate, unfinished "skills" feature sitting uncommitted in the same working tree: `server/src/modules/index.ts` and `server/src/db/rows.ts` already had uncommitted skills-module registration lines before the conventions task touched them (client's `nav.ts` had the identical problem ‚Äî see `client/INSIGHTS.md`). Fix pattern: `git show HEAD:path/to/file > /tmp/clean.ts`, hand-edit `/tmp/clean.ts` to add ONLY your change on top of real HEAD content, `cp /tmp/clean.ts path/to/file && git add path/to/file` and commit that ‚Äî then re-apply your original full edit (your change + the pre-existing WIP together) to the working tree afterward, uncommitted, so `git diff HEAD -- path/to/file` shows only the pre-existing WIP residual and the file stays fully correct for local testing. Verify with `git show --stat <commit>`, not just `git status` after the fact; `git reset --soft HEAD~1` + redo is the recovery move if caught late.

- **Cleaner than the `/tmp` cp-back dance above: stage a hand-built blob into the index and DON'T touch the working tree.** When a shared file (`modules/index.ts`, client `hooks/index.ts`) has BOTH your change and unrelated WIP lines you must exclude, pathspec can't help (it stages the whole file). Instead: `grep -vxF -e '<wip line>' ‚Ä¶ path > "$SCRATCH/staged"` (keep yours + HEAD, drop the WIP lines) ‚Üí `blob=$(git hash-object -w "$SCRATCH/staged")` ‚Üí `git update-index --cacheinfo 100644,$blob,path`. This puts your scoped content in the index while the working file keeps its WIP for local dev ‚Äî then commit with a **BARE `git commit`** (a pathspec here would re-stage the full working file and undo the surgery; this is the one case bare-commit is correct ‚Äî you curated the index by hand, so first confirm `git diff --cached` shows only your lines and nothing else is staged). Used to land the Smart Diff registration lines in `modules/index.ts` while leaving the uncommitted Intent-Layer `intent,` line out (2026-07-08). Don't wrap this in a bash *function* in the Bash tool ‚Äî the zsh wrapper lost PATH/`mktemp` mid-function; inline the commands.

- **A `FeatureModelId` registry entry can default to a provider you have no key for.** `FEATURE_MODELS` (`vendor/shared/contracts/platform.ts`) hardcodes a default `provider`/`model` per feature ‚Äî e.g. `conventions` ‚Üí `openai`/`gpt-5.4` ‚Äî independent of what's actually in `~/.devdigest/secrets.json`. A partial local setup may only have `OPENROUTER_API_KEY`, so calling an unoverridden feature fails at call time with `{"error":{"code":"config_error","message":"OPENAI_API_KEY is not configured"}}`. Fix: `PUT /settings` with `{"feature_models": {"<id>": {"provider": "openrouter", "model": "<a real openrouter model>"}}}` before exercising that feature. Resolution goes through `resolveFeatureModel(container, workspaceId, id)` (`modules/settings/feature-models.ts`), which falls back to the registry default only when no override row exists. A `routeModel(...)` helper is referenced in at least one planning doc (`docs/conventions-extractor-plan.md`) but does **not exist** in this codebase.

- **A new `NOT NULL text` column with no `DEFAULT` on an existing table breaks the migration the moment the table has any rows** ‚Äî drizzle-kit happily generates a bare `ALTER TABLE ... ADD COLUMN category text NOT NULL` (no default) if the schema field is `text('category').notNull()` with nothing else; it applies fine against an empty dev table but fails on any environment where the table is already seeded. `status` on the same table shipped correctly (`.notNull().default('pending')`) ‚Äî always add a `.default(...)` (or split into add-nullable ‚Üí backfill ‚Üí set-not-null across migrations) for any `NOT NULL` column added to a table that might already have rows.

- **API serving stale code despite edits ‚Üí check for duplicate `tsx watch src/server.ts` processes.** Multiple `pnpm dev` instances can accumulate; only one binds `:3001` and the others' children may keep serving OLD code (or a killed watcher leaves an orphaned child on the port). Symptom: a route change has no effect even though the file on disk is correct. Fix: `lsof -nP -iTCP:3001 -sTCP:LISTEN` to find the real owner, `kill` all `tsx watch` parents + the listening child, confirm `:3001` is free, then start one fresh `pnpm dev`.

- **`docker exec devdigest-postgres psql -U postgres ‚Ä¶` fails with `FATAL: role "postgres" does not exist`.** The container has no default superuser role ‚Äî the DB and role are both `devdigest` (see `DATABASE_URL` in `server/.env` / `docker-compose.yml`). Use `psql -U devdigest -d devdigest`. Useful for ad-hoc checks like confirming an applied migration: `select id, created_at from drizzle.__drizzle_migrations order by id desc;`.

## Codebase Patterns

- **For a "free"/token-free feature that composes existing PR data, read `pr_files` via `container.reviewRepo.getPrFiles(prId)` ‚Äî NOT the `GET /pulls/:id` detail path.** The pulls detail route (`modules/pulls/routes.ts`, the ~170-210 block) is NOT a pure DB read: when a GitHub token is configured it re-fetches the PR from GitHub and DELETE+re-INSERTs the `pr_files` rows on every request. So reusing "the pull's files" by calling that route/service would silently trigger a network round-trip (and DB writes) ‚Äî invisibly breaking a "no new model call / no network / free" acceptance criterion. `getPrFiles` is a plain `select ... where prId` (see `repository/pull.repo.ts`), and `reviewRepo` also exposes `getPull` (workspace-scoped 404 guard) + `reviewsForPull`. The new `modules/smart-diff/service.ts` composes entirely from those three ‚Äî zero `container.llm()`, zero GitHub ‚Äî which is the whole point of Smart Diff (the expensive LLM call already happened in the reviewer; this only regroups). Note smart-diff intentionally takes the SINGLE newest review (`reviewsForPull(prId)[0]`) for its `finding_lines`, which can differ from the PR-list's "current findings PER AGENT" semantics (see the `currentReviewIdsByPr` entry below) ‚Äî a PR reviewed by multiple agents will badge only the newest run's findings.

- **`ReviewRepository` (repository.ts) wraps `run.repo.ts`** ‚Äî both files define `completeAgentRun`. When adding a new field to the values type, update BOTH files: `repository/run.repo.ts` (the implementation) and `repository.ts` (the wrapper used by `run-executor.ts`).

- **`GET /repos/:id/pulls` returns `Promise<PrMeta[]>` with NO response Zod schema** (only `params: IdParams`). Adding a field to `PrMeta` is therefore a pure type change ‚Äî the object is serialized as-is, nothing strips unknown keys. The list popover's `latest_findings` (full `Finding[]`) is built in the SAME findings query that computes the severity counts (`fullFindingsByReview` alongside `findingsByReview`), so previewing findings on the list needs no extra request. `severity`/`category`/`kind` are `text` columns ‚Üí cast to the `Finding` enum types (`as Finding['severity']`).

- **PR-list FINDINGS + SCORE aggregate across the latest review PER AGENT, not the single newest review** (`currentReviewIdsByPr` keyed by `prId:agentId`, newest-first ‚Üí first seen wins). Re-runs create new `reviews` rows, so a PR that one agent re-ran clean while another still flags issues must still show those issues ‚Äî "current open findings" semantics (product-confirmed: clean latest reviews ‚áí show "‚Äî"). PR `score` is the MIN across those current per-agent reviews (worst agent), so the score stays coherent with the findings shown. The `/pulls/:id/reviews` endpoint, by contrast, returns ALL review rows (full history) ‚Äî so the detail page shows superseded findings the list intentionally omits.

- **The PR-list aggregation (score + findings + cumulative cost) lives in `modules/pulls/review-meta.repo.ts`, NOT inline in `routes.ts`.** `loadPrReviewMeta(db, priceBook, prIds)` runs the 3 IN-queries + JS grouping and returns `Map<prId, PrReviewMeta>`; the route handler just maps rows through it. Keep new list-aggregation logic there ‚Äî `routes.ts` is intentionally thin (per `server/CLAUDE.md`: routes thin, data access in repositories). The route still does the GitHub sync + base PR select directly; that's pre-existing.

- **Run-cost fallback is centralized in `PriceBook.resolve(costUsd, model, tokensIn, tokensOut)`** ‚Äî returns the actual API cost when stored, else `estimate(...)` for pre-migration rows, else null. Don't re-inline the `costUsd ?? estimate(...)` ternary; the list (`review-meta.repo.ts`), run history (`reviews/service.listRuns`), and trace (`reviews/service.getRunTrace`) all go through `resolve` so they stay consistent.

- **Drizzle transactions + row locking ARE available and now used** (first site: `modules/skills/repository.ts`). `this.db.transaction(async (tx) => { ... })` ‚Äî do all writes on `tx`, return value commits, throw rolls back. For a read-modify-write on a counter (e.g. bumping `skills.version` + inserting the matching `skill_versions` snapshot), lock the row inside the txn with `.select()....where(...).for('update')` so concurrent edits serialize instead of both reading version N and colliding on `(skillId, version)`. Pattern to AVOID: reading via a non-locking helper (`getById`) then updating ‚Äî that's the race. Also: `.for('update')` is drizzle's `SELECT ... FOR UPDATE`; prefer it over `onConflictDoNothing`, which silently swallows the lost write and hides the divergence.

### 2026-07-04 ‚Äî pr-self-review re-run found a previously "fixed" CRITICAL was not actually fixed
- **Patching a bad migration with a LATER migration doesn't fix it ‚Äî the bad one still runs first.** Commit `ccba0dd` believed it fixed the `category text NOT NULL` no-default bug (see the "A new `NOT NULL text` column..." entry above) by adding `.default('uncategorized')` to `schema.ts` and generating a new migration `0012` containing only `ALTER COLUMN category SET DEFAULT 'uncategorized'`. It never regenerated `0011` itself, which still has the bare `ADD COLUMN category text NOT NULL` with no default. The correct fix for a not-yet-shipped migration is to delete the patch migration and **regenerate the original one** from the now-corrected schema, not bolt a follow-up `ALTER COLUMN` onto it.
- **A `pr-self-review` report's `outcome: "fixed"` / `fixCommit` annotation is a claim, not a proof ‚Äî re-verify it on every re-run, don't just carry it forward.** The prior report (headSha `1d2bfb6`) recorded this bug as `"outcome": "fixed"` and the branch went on to a `"verdict": "pass"`. Two commits later, re-running the skill's deterministic migration-safety check (reading the actual `.sql` files, not trusting the old report) found the bug still live. Treat every prior "fixed" finding as a hypothesis to re-check against current file contents, especially for migration files, which are easy to "fix" at the wrong layer (schema vs. the already-generated SQL).

### 2026-07-08 ‚Äî WebFetchClient SSRF adapter (Intent Layer external plan/spec fetch)
- **`new URL(...).hostname` keeps the `[...]` brackets around an IPv6 literal, but `dns.promises.lookup()` requires the bare form.** `new URL('http://[::1]/').hostname === '[::1]'` ‚Äî passing that straight to `dns.lookup()` throws `ENOTFOUND` for EVERY IPv6-literal host, not just blocked ones. A naive "DNS lookup fails ‚Üí fail closed (block)" guard (the right default for genuine failures) would therefore block ALL legitimate public IPv6 hosts too, since the bracket form never resolves. Fix in `adapters/http/web-fetch.ts#isHostnameBlocked`: strip a leading `[`/trailing `]` before calling `lookup()`. Verified with a scratch script: `http://[::1]/` still blocks (loopback), and `http://[2606:4700:4700::1111]/` (Cloudflare) now gets past the SSRF guard and only fails at the actual network call (sandbox has no egress), not at the guard.
- Added `WebFetchClient`/`WebFetchResult` to `vendor/shared/adapters.ts` (byte-identical addition mirrored into `client/src/vendor/shared/adapters.ts`), `HttpWebFetchClient` in `adapters/http/web-fetch.ts`, `container.webFetch` getter, and `FakeWebFetchClient` in `adapters/mocks.ts`. `externalFetchEnabled` config defaults to `true` (`EXTERNAL_FETCH_ENABLED !== 'false'`) ‚Äî safe only because of the adapter's SSRF guard (scheme allowlist, DNS-resolved IP-range block covering the resolved addresses of every hop, manual redirect re-validation capped at 3 hops, streamed size cap, `AbortSignal.timeout`). No `*.test.ts` was added per this task's scope (mock is for other agents' future test use); a follow-up task should cover this adapter with real unit tests (SSRF-guard correctness is security-critical and deserves direct test coverage, not just the manual sanity check done here).

### 2026-07-08 ‚Äî Intent Layer convergence: path-traversal guard + references.ts split
- **`container.git.readFile` (`adapters/git/simple-git.ts:129-130`) has NO path confinement of its own** ‚Äî it's a bare `readFile(join(clonePathFor(repo), path), 'utf8')`. The old `resolvePlanRefs` (formerly in `signals.ts`) extracted repo file paths straight out of the attacker-controlled PR body via `PLAN_PATH_RE` (whose char class allows `.` and `/`) and passed them directly to `readFile` ‚Äî a PR body containing `docs/plans/../../../../../../etc/passwd` would read arbitrary local files and feed the content to the LLM. Any future caller of `container.git.readFile` with a body-/user-derived path needs its OWN guard before the call ‚Äî the port itself does not protect you. Fixed by adding `isSafeRepoPath` (`modules/intent/references.ts`) applied to every candidate path before `readFile`: normalize with `path.posix.normalize`, then reject if absolute, equal to `..`, starts with `../`, or any segment (`split('/')`) is `..`. Plain relative paths like `docs/plans/intent-layer.md` pass through unchanged.
- Moved link/reference resolution out of `signals.ts` into a dedicated `modules/intent/references.ts` (`resolveReferences`, renamed from `resolvePlanRefs`) to match the reference layout, and added a third source kind ‚Äî external (non-GitHub) URLs ‚Äî routed through the already-existing `container.webFetch.fetch(url)` (SSRF-guarded centrally there; never call global `fetch` from a module). Detection: extract all `https?://` URLs from the body, then exclude any that match `ISSUE_URL_RE` (already handled as GitHub issue/PR links in kind (b)) before capping to 3 and fetching.
- **The `..`-string guard is NOT enough ‚Äî committed symlinks bypass it; confine at the sink with `realpath`.** An attacker's cloned PR repo can ship a git-tracked symlink `docs/plans/x.md ‚Üí ../../../../secrets.json`; that path has no `..` so `isSafeRepoPath` passes, and `readFile` follows the link out of the clone root (clone root `~/.devdigest/workspace/<owner>/<name>` ‚Üí the symlink lands exactly on `~/.devdigest/secrets.json`, the SecretsProvider store, whose content then goes to the LLM). Found by the adversarial security review. Fixed defense-in-depth INSIDE `simple-git.ts#readFile`: `realpath` both the clone root and the requested target, and throw unless `targetReal === rootReal || targetReal.startsWith(rootReal + sep)`. `realpath` on a missing file throws ‚Üí existing `.catch(() => null)` callers are unaffected; only symlink-escapes newly get skipped. Lesson: for a repo you didn't author, string-normalizing the path is insufficient ‚Äî resolve the real path and confine it, at the port so every caller is covered.
- **`EXTERNAL_FETCH_ENABLED` now defaults to `false` (opt-in), NOT `true` (this corrects the earlier note above that said `!== 'false'`).** Even with the SSRF guard fixed to fail-closed, a residual DNS-rebinding TOCTOU remains: the guard resolves the host with `dns.lookup`, then global `fetch` resolves AGAIN independently (no IP pinning), so a fast-rebinding attacker domain can answer "public" to the guard and "127.0.0.1"/"10.x" to `fetch`. Properly closing it needs connection pinning via an `undici` dispatcher with a custom `connect.lookup` ‚Äî but `undici` is not a dependency of `server/`, and we didn't add one. So the safe default is opt-in: `externalFetchEnabled = EXTERNAL_FETCH_ENABLED === 'true'`. To make `true` safe by default, add `undici` and pin the validated IP across the request + every redirect hop.

### 2026-07-21 ‚Äî Cheap incremental pr-self-review after a trivial follow-up commit

- **`pr-self-review`'s gate hashes the WHOLE `main...HEAD` diff, so appending even a one-line commit to an already-reviewed branch makes the hook call the existing report "stale" and re-block push** ‚Äî but that doesn't mean the full multi-file, multi-lens review has to be redone. `.claude/pr-self-review-report.json` records the `headSha` its findings were computed against; diffing that recorded `headSha` to current `HEAD` (`git diff <old-headSha>..HEAD --name-only`) isolates exactly what changed since the last **pass** verdict. Dispatched lenses only for that incremental file set (here: one sentence added to `INTENT_SYSTEM` in `modules/intent/prompts.ts`), carried forward the prior report's untouched findings verbatim, then wrote a new report with the updated `headSha`/`diffHash`. Confirmed correct because the prior report's `diffHash` had been computed from `main...a2a035c`, and the only files touched since were a docs file (skipped, non-code) and the one prompt-string line. Re-running all ~10 lenses across ~60 files for a 1-line follow-up would have been pure waste.
- **When adding a new trusted instruction sentence to an LLM system prompt that also contains an "everything below is untrusted" boundary sentence (e.g. `INTENT_SYSTEM`'s wrapUntrusted-fenced sections), insert the new instruction BEFORE that boundary sentence, not after.** Three independent lenses (onion-architecture, typescript-expert, security) confirmed no issues with the placement chosen for the new "always respond in English" sentence in `prompts.ts` specifically because it stayed inside the trusted-instruction portion of the string, ahead of the "follow only these system instructions" guardrail ‚Äî keeping the constraint unambiguously part of the model's own instructions rather than adjacent to attacker-influenceable content.
