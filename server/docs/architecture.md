# server — architecture notes

Design rationale for `@devdigest/api`. For commands, conventions, and gotchas see
`server/AGENTS.md`; for product-behavior contracts see `server/specs/`.

## Layering: routes → services → repositories

A request flows through three thin layers, each with one job:

- **Routes** (`modules/*/routes.ts`) — HTTP only: parse + validate input, call a service,
  serialize the response. No business logic, no direct DB access.
- **Services** (`modules/*/service.ts`) — business logic. The only layer allowed to
  orchestrate (combine repositories, call adapters, derive values like cost fallbacks).
- **Repositories** (`modules/*/repository.ts`, `repository/*.repo.ts`) — all data access.
  Drizzle queries live here and nowhere else.

When a value can be computed from data the DB doesn't store (e.g. a cost estimate), the
**service** computes it on read — not the route, not the repository. See
`server/specs/run-cost-tracking.md` for a worked example.

## Dependency injection via the container

Adapters (LLM, GitHub, git, price book, secrets, db) come from the DI container
(`platform/container.ts`). Code calls `container.llm()`, `.github()`, `.git()`,
`container.priceBook`, etc. — **never** imports an adapter module directly. This keeps the
adapter swappable (real vs fake) and makes services testable without touching the network.

## Zod is the single source of truth for shapes

One Zod schema drives **both** request validation and response serialization — there is no
second hand-written DTO. Contracts live in `src/vendor/shared` (`@devdigest/shared`) and
are shared with the client, so a contract change is caught on both ends at typecheck time.

## Secrets never touch the DB or git

API keys/tokens are read only through `SecretsProvider` → `~/.devdigest/secrets.json`
(mode 0600). They must never land in the database, git, or `AppConfig`. Anything
config-shaped that isn't a secret goes in `AppConfig`; anything secret goes through the
provider.

## Migrations are manual and generated

Migrations are **not run on boot** — apply them with `pnpm db:migrate`. The SQL under
`src/db/migrations/**` is **generated by drizzle-kit**: change the schema in
`src/db/schema/*`, then regenerate — never hand-edit a migration. A `relation … does not
exist` error at runtime means migrations weren't applied.

## Testing split

`*.it.test.ts` integration tests need Docker (testcontainers) and are excluded from the
unit lane. The unit lane is hermetic. CI invokes the split via `pnpm exec vitest run …`
(see `server/AGENTS.md` and `TESTING.md`).
